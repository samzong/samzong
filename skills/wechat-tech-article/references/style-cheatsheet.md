# WeChat Writing Style Cheatsheet (Enforced)

这是一份“速查表”，用于让 AI 稳定复刻你的公众号味道；完整版请看：

- `skills/wechat-tech-article/references/style-guide.md`

---

## 作者人设（一句话）

工程师/产品经理视角的开源作者：爱工具、重效率、讲工作流；写法像“我亲自踩坑 + 立刻给你可复用步骤”。

---

## 结构硬规则（默认强制）

- 第一行必须是 `# 标题`
- 标题后 10–15 行内必须出现：
  - 入口链接（GitHub/下载/在线体验其一）
  - 一句加粗 `**tagline**`（是什么 + 适合谁 + 解决什么）
- 文章正文只用 `#` / `##` 标题（禁止使用 `###` 及更深层级）
- 不使用 `1/2/3`、`**01**/**02**` 这类数字锚点
- 至少 2 个可复制代码块（bash/yaml/toml/json 任选），并配解释
- 段落短：一段不超过 3–4 行；能拆就拆

---

## 排版签名（读者一眼认出你）

- `**加粗**` 密集，用来标关键词/结论/段落名
- 列表很多：功能/步骤/要点都用 `-` 或编号列出
- 代码块像“可复制说明书”，不写装饰性代码
- 图是叙事工具：每张图出现前后都要一句解释（它展示/证明什么）
- 用 `---` 做模块化分隔，增强扫读体验

---

## 两种模式（只决定内容组织）

### 模式 A：开源项目/工具发布（公众号型）

**读者期待**：入口 + 截图 + 功能清单 + 30 秒跑起来。

**稳赢信息顺序**：
- 标题 + 首图 + `**tagline**`
- 入口链接（开头 10–15 行内）
- 缘起/痛点清单（3–5 条）
- 产品定义一句话（设计原则/边界）
- 功能介绍（3–7 条，写“收益 + 触发方式”）
- 如何使用（最短安装路径 + 最短示例）
- 技术说明/设计选择（可选但很加分）
- 写在最后：后续规划 + 轻 CTA（欢迎反馈/Star/试用）

**常用标题钩子**：
- “我开源了一个……”
- “不会吧？你还在……？”
- “这个开源插件帮助我从 A 迁移到 B”
- “我把 X 做成了一个顺手的 CLI”

### 模式 B：技术科普/架构深挖（技术博客型）

**读者期待**：概念对齐 + 机制拆解 + 适用边界 + 关键结论。

**推荐结构**：
- 写在前面：为何关注/阅读建议/系列规划（可选）
- 核心问题与挑战（3–6 条）
- 核心架构（图 + 数据流）
- 关键组件逐个讲：职责/接口/为什么这么设计
- 取舍与适用场景：SLO/TCO/兼容/成本
- 小结：3 条 bullet 收尾

---

## 标题工坊（爆款但不虚）

**硬门槛**：不能承诺无法证明的收益（“10 倍”这类除非你有数据/基准）。

**输出**：8 个候选（分组），每个附“适合读者 + 主打收益点”，给 Top 2 推荐理由。
