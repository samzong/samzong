## 你的“作者人设”（一句话）

偏工程师/产品经理视角的开源作者：爱工具、重效率、讲工作流；写法像“我亲自踩坑 + 立刻给你可复用步骤”，并且愿意把技术深挖到底（云原生/AI Infra/CLI 方向尤多）。

## 风格指纹（读者一眼能认出的点）

### 1) 叙事方式

- 先抛“真实痛点/场景”：工作里遇到什么麻烦、哪里不顺手、为什么要做/要换工具。
- 快速亮底牌：你做了什么（开源项目/插件/CLI/方案），一句话说明它解决了什么。
- 以“可操作”为主：大量命令行、配置片段、步骤列表，读者看完能立刻复现。
- 适度的自嘲/口语化：不端着，但也不轻浮；偏“同事聊天式技术分享”。

### 2) 结构套路（非常稳定）

常见骨架是：

1. 标题（强钩子：疑问句/反差/效率提升/“我做了个 X”）
2. 一张引导图（或首屏图）
3. 背景与动机（我为什么要做/要换/要研究）
4. 核心能力/功能列表（分点、分段、分小节）
5. 用法/安装/部署（给可复制命令）
6. 对比/取舍/关键结论（让读者知道何时该用）
7. 写在最后/后续规划（收尾 + 轻 CTA）

你经常用 `**01** / **02** / **03** / **04**` 作为大段落分隔符，配合小标题（有时再用 `2.1 / 2.2` 这类层级编号）。

### 3) 语言风格

- 以第一人称为主（“我/我在/我做了/我习惯”），偶尔对读者用“你/您”。
- 中英混写偏自然：关键名词、工具名、协议名、命令保留英文，解释用中文。
- 句子不追求文学感，追求“清晰 + 快”：短段落很多，段落之间用空行拉开节奏。
- 强调词高频：效率、顺手、一键、工作流、开源、全平台、CI/CD、迁移成本、兼容、隐私、安全、核心能力、后续规划。

### 4) 排版与标记（你的“排版签名”）

- 大量加粗：用 `**...**` 标出关键词、结论、段落名。
- 代码块密集：Shell/YAML/配置/伪代码经常出现，且更像“可复制说明书”。
- 列表很多：用 `-` 或编号列出功能/步骤/要点。
- 偶尔引用块：用 `>` 放吐槽、补充说明或约束。
- 横线分隔：用 `---` 把段落分成“模块化阅读”。
- 图片很多：通常插在段落之间做“视觉换气”，并且图片本身是叙事的一部分（展示 UI、效果、图表、Logo）。

## 两种“写作模式”（你实际上在切换）

### 模式 A：开源项目/工具发布（偏公众号风）

特征：

- 钩子强（标题就带情绪或反问）
- 讲一个“为什么做”的故事，然后立刻给地址、给截图、给功能清单
- 读者读完最想要的是：安装方式 + 关键功能 + 能不能马上试

推荐结构：

1. 标题：反问/惊讶/利益点（例如“不会吧…你还在…？”、“我开源了…工具”）
2. 一句话定位：这是什么 + 给谁用 + 解决什么
3. 痛点：迁移成本/效率/隐私/复杂度
4. 功能清单：用 5-10 条 bullet 写“可感知收益”
5. 怎么用：安装/运行/命令示例（越可复制越好）
6. 设计选择：为什么这么设计（不接管终端、兼容现有配置、数据本地等）
7. 写在最后：欢迎反馈/Star/后续计划

#### 项目介绍的“信息顺序”（你的固定打法）

你写“一个公众号文章对应一个开源项目”时，信息出现的顺序非常稳定，读者会感觉像在看一份「发布说明 + 使用手册 + 设计复盘」的组合：

1. **首屏定调**：标题 + 首图 + 一句英文/中文 tagline（通常加粗），快速贴标签（平台/风格/定位）。
2. **立刻给入口**：在开头 10～15 行内给 GitHub / WebStore / 在线体验链接（“项目地址/仓库链接”基本不拖）。
3. **缘起/痛点清单**：用 3～5 条列出“不爽点”（旧方案的无关功能、跨平台性能、迁移成本、隐私/账号、操作繁琐等）。
4. **产品定义一句话**：把你的“设计原则”写成一句可引用的话（例如“不接管终端/数据本地/克制 UI/兼容原有配置”）。
5. **功能介绍（强分段）**：
   - 用 `**01**/**02**/...` 做大段落锚点
   - 功能点用编号或 bullet，倾向写“收益 + 触发方式（快捷键/命令）”
   - 很喜欢补「版本演进」：`v0.x/v2.x` 按版本列能力，让读者看到持续维护
6. **如何使用（可复制为王）**：Homebrew / Docker / 插件商店 / Release 下载，配上最短路径命令。
7. **技术说明（可选但加分）**：用一小段把关键方案讲清楚（例如 Manifest V3、iframe、CI/Action、协议/组件等），塑造“懂实现”的可信度。
8. **写在最后 + CTA**：感谢用户/欢迎反馈/求 Star/后续规划/引流到其他文章或公众号关注。

#### 你常用的标题钩子（项目发布向）

- “我开源了一个[工具/软件]”
- “不会吧？你还在[做低效的事]？”
- “这个开源插件帮助我从[A]迁移到[B]”
- “我把[某个操作]做成了一个顺手的 CLI”

#### 版本更新文的“固定结构”（以 TabBoost 更新文为代表）

- 开头先交代：发布后用户反馈/感谢 + 发现问题
- 插入一个“重要通知/渠道变化”（例如国内可安装、镜像渠道）
- `**01** 功能更新`：以收益点口吻列新增能力（常带快捷键、场景）
- `**02** 故障修复`：引用用户反馈（`>`）+ 逐条说明已修复
- 末尾再次引导：下载链接/Star/继续反馈

#### 项目介绍写作“素材清单”（写之前先准备）

- 一句话定位（可当 tagline）：**[产品名] 是一个 [平台/形态] 的 [类别]，解决 [痛点]**
- 3～5 条“我为什么要做它”（最好能对标旧方案缺点）
- 3～7 条“核心能力”（每条尽量是：收益 + 触发方式）
- 最短安装路径（Homebrew/Docker/WebStore/Release 任一，能 30 秒跑起来）
- 对比对象（Arc/Chrome、pip/Poetry、手写 config/工具管理等），给一句结论
- 可信度补丁（开源/本地存储/不接管终端/兼容现有配置/隐私）
- 一句话后续规划（让读者知道会持续更新）

### 模式 B：技术科普/架构深挖（偏技术博客风）

特征：

- 更正式、层级更清晰（`##`、`###`、表格、对比结论）
- 先定义问题与背景，再逐步拆系统（组件、数据流、关键机制）
- 读者读完最想要的是：理解框架 + 复用思路 + 关键术语对齐

推荐结构：

1. 标题：明确主题（例如“X：Y 介绍（一）”、“新范式：玩转 Z”）
2. 写在前面：你为何关注、系列文章规划、阅读建议
3. 核心问题与挑战：用 3-6 条列表定边界
4. 核心架构：图 + 数据流（或伪图/伪代码/CRD）
5. 关键组件：逐个讲职责、接口、为什么这么设计
6. 取舍与适用场景：SLO/TCO/兼容/成本
7. 小结：关键结论（最好能用 3 条 bullet 结束）

可直接复用的句式模板（用你自己的内容替换方括号）：

- “在实际工作中经常需要[场景]，但[现状/旧方案]带来[痛点]。”

- 每个大段落要有“可扫读的锚点”：加粗小标题 + 列表/代码块。
- 每隔 3-6 段插入“结构变化”（图/代码/列表/引用）让读者不疲劳。

## Markdown 规范（写作时就按这个来）

### 标题与层级

- 第一行必须是 `# 标题`
- 大章节用 `##`，小章节用 `###`
- 可选：在大章节前插入 `**01** / **02**` 作为“公众号式分段锚点”

### 强调与可读性

- 关键术语、结论、按钮名、命令名：用 `**加粗**`。
- 术语第一次出现：给一句解释，不要默认读者全懂。

### 代码块

- 命令行：用 ```bash
- 配置：用 ```yaml / ```toml / ```json
- 代码示意：允许伪代码，但要标注“伪代码/简化示意”

### 链接与资源

- 项目地址可以直接给裸链接或“项目地址：...”
- 如果引用数据或结论来自外部：给出处链接（别让读者“信我就完了”）

### 图片

- 图片是“叙事工具”，不是装饰：每张图出现前后都应有一句解释（它证明什么/展示什么）
- 建议避免生成 `data:image/svg+xml` 这类占位图片链接（它会污染 diff、也会干扰后续文本分析）

## 发布前 Checklist（写完 2 分钟自检）

- 标题是否一眼看出“收益点/反差/痛点”？（别太平）
- 开头 10 行内是否讲清楚：你做了什么、解决什么、给谁用？
- 是否给了“最短路径的可复制步骤”？（安装 + 运行 + 一个真实例子）
- 是否有 1 个对比或取舍说明？（何时该用/不该用）
- 结尾是否有“下一步计划”或“欢迎反馈”的收束？
- 命令/配置是否可复制、格式是否正确？（避免拼接在一行）
